interactive-scenario-modeller — LLM 7 Instructions (Agent-Ready)

Purpose
- Use this library to model multi-year entity stock evolution (Filter → Prioritise → Transform).
- Designed for agent-orchestrated visual analytics: keep data-agnostic, modular, and slim.
- Core engine handles the simulation loop; domain expertise is provided via opt-in plugins.

Core Concepts: The Trinity P-F-T
1) Filter (Eligibility)
- Determines which entities/features are candidates for a transformation in a given step.
- Context: `(entity, context) => boolean`.

2) Prioritise (Ranking)
- Determines the order of processing when resources (budget, capacity) are limited.
- Context: `(e1, e2, context) => number` (Sort logic). Supports "random" or MCDA plugins.

3) Transform / Upgrade (Effect)
- Applies the change and returns metric deltas.
- Context: `(entity, context) => MetricDelta`.

Library Architecture (Modular Imports)
- Core: `interactive-scenario-modeller` (SimulationRunner, Intervention, types).
- Plugins: `interactive-scenario-modeller/plugins/<category>` (grid, financial, geographic, etc.).
- Global Registry: Use `registerPredicate` and `registerPlugin` for serializable scenarios.

Entity & Facet Data
- Entities: Generic objects with an `id` and optional `geometry`. Avoid hardcoding `uprn`.
- Facets: Use `arrayAdapter(data)` for in-memory collections or custom adapters for streaming/SQL.
- Geometry: First-class support for GeoJSON logic (`within`, `distanceTo`) via `geographic` plugins.

Context (SimulationContext)
- `context.year` / `context.step`: Current timestep.
- `context.resources`: Global bank for cross-intervention constraints (`consume`, `has`).
- `context.state`: Shared mutable state across the entire simulation run.
- `context.random()`: Deterministic RNG for reproducible "what-if" outcomes.

Recommended Agent Workflow for Dashboards
1) Initialize: Create `SimulationRunner` and register domain plugins.
2) Setup: Define `Intervention` instances using string-based plugin references for UI-driven config.
3) Run: Execute `runner.run()` or `intervention.simulate()`.
4) Visualize: Map `result.metrics` to charts and `toGeoJSON(intervention.simulatedFacet)` to maps.

Plugin Categories Available
- geographic: Spatial clustering, regional budget splits, urban/rural strategies.
- optimization: Multi-criteria prioritiser (AHP/MCDA), carbon target constraints.
- transport: EV load interaction, corridor constraints.
- financial: Budget spend tracking, financing models (Capex/Opex/PPA).
- grid: Substation capacity gates, sequential enablement (pre-requisites), headroom.
- risk: Monte Carlo and Sensitivity analysis runners.

Implementation Tips
- Performance: The runner is optimized for 50k+ entities in-browser.
- Memory: Interventions write deltas back to the facet; successive interventions see updated states.
- Resource Bounds: Always use `context.resources.consume()` for shared pools (e.g., "Max 500 installs per year").
- Schema Mapping: All plugins support option mapping (e.g., `buildingIdField: "asset_tag"`).

How to Build Scenarios Reliably
- Keep context.state explicit and versioned by scenario if needed.
- Use deterministic prioritisation where reproducibility matters.
- Use string plugin refs in persisted scenario configs.
- Keep one plugin focused on one responsibility (constraint OR upgrade).
- Write init hooks that set only missing state defaults unless override is intended.

How to Expand the Plugin Ecosystem

A) Design Checklist (before coding)
1) Define plugin kind: constraint, upgrade, prioritiser, etc.
2) Define minimal required building fields.
3) Define required state keys and optional defaults.
4) Define output metric contract for downstream reporting.
5) Choose clear plugin name and export key (usually "constraint" or "upgrade").

B) Implementation Steps
1) Create plugin file in appropriate domain folder:
   - src/plugins/financial/
   - src/plugins/social/
   - src/plugins/policy/
   - src/plugins/grid/
   - or new domain folder if justified
2) Implement factory function:
   - createXPlugin(options) => PluginRegistration
3) Add manifest:
   - name, version, kind, description, entry, compat, trusted, exports
4) Add handler logic:
   - constraint(building, context) => boolean
   - upgrade(building, context) => metrics object
5) Export from domain index.ts and from src/index.ts
6) Add tests:
   - plugin registration test
   - positive behavior test
   - edge-case test(s)
   - intervention simulate integration test via string reference
7) Add docs:
   - README plugin listing
   - use-case cookbook entry in docs/scenario-modeller-use-cases.md

C) Quality Rules
- Keep plugins pure unless state writes are intentional and documented.
- Validate/sanitize numeric inputs (NaN, strings, missing fields).
- Support configurable field/state keys for interoperability.
- Keep default behavior predictable and simple.
- Avoid hidden side effects across unrelated state keys.

D) Naming and Reference Conventions
- Name plugin by domain + behavior (example: "grid-substation-capacity-gate").
- Use explicit export names in refs ("plugin-name:constraint", "plugin-name:upgrade").
- For bundled installs, always use returned exportRef values instead of rebuilding strings manually.

Persistence and Interoperability Guidance
- Persist intervention config with string refs for filter/prioritise/upgrade.
- Persist scenario state keys explicitly (budgets, policies, capacities, completion maps).
- Reinstall bundles/plugins before replaying persisted scenarios.

Large Dataset Guidance
- Current simulation path is in-memory per intervention/year; memory pressure grows with:
  - row count,
  - number of columns copied into buildings,
  - number of interventions run in sequence.
- For large datasets, always use `setupBuilding` to project only required fields.
- Prefer narrow plugin field contracts and avoid carrying unused columns in scenario state.
- Run sensitivity/Monte Carlo on filtered subsets first, then scale to full stock.
- Use scenario template defaults and bundle refs to avoid expensive per-run setup logic.

Large Dataset Operational Playbook (Required for big runs)
1) Profile first
- Compute row count and file size before running scenarios.
- Treat datasets around >=50k rows or >=25MB as “large” for default in-memory workflows.

2) Minimize row payload in setup
- In `setupBuilding`, include only fields required by active filter/prioritise/upgrade plugins.
- Avoid copying full CSV rows into simulation buildings.
- Normalize key numeric fields once in setup; avoid repeated parsing in predicates.

3) Stage execution
- Stage A: smoke test on 1k–5k sampled rows.
- Stage B: medium run on 10k–25k rows.
- Stage C: full run after checking memory/runtime trends.
- Keep plugin stack identical across stages to preserve behavior comparability.

4) Control intervention complexity
- Prefer fewer interventions per run; chain only when needed.
- Use `SimulationRunner` for explicit sequencing and inspect intermediate outputs.
- Avoid broad `includeUntouched` outputs on very large runs unless required.

5) Risk-helper strategy at scale
- Use `runSensitivityAnalysis` with narrow parameter grids first.
- Use `runMonteCarloAnalysis` with small run counts initially (e.g., 10-25), then increase.
- Do not combine large parameter grids and high Monte Carlo runs in the same first pass.

6) State hygiene
- Keep `context.state` small and structured.
- Store only aggregate counters/maps required for constraints and metrics.
- Avoid storing whole-building arrays in state during simulation.

7) Validate before trusting results
- Track per-year selected counts and key aggregates (budget spent, capacity loads, carbon totals).
- Compare sampled-run distributions against full-run outputs for drift detection.
- Re-run with fixed assumptions to confirm deterministic behavior where expected.

8) Winchester-specific recommendation
- For `winchester_building_stock.csv` (~72k rows, ~53MB), use staged execution and field projection by default.
- Run one baseline full scenario first, then add plugin families incrementally (optimization -> geographic -> timeseries -> risk).

Common Pitfalls and Fixes
- Pitfall: plugin not found at runtime
  - Fix: ensure registerPlugin/install bundle runs before creating intervention
- Pitfall: no buildings selected unexpectedly
  - Fix: inspect filter constraints and state defaults; log candidate scores
- Pitfall: numeric fields stored as strings
  - Fix: parse numbers safely inside plugin logic
- Pitfall: policy validator expected to throw globally
  - Fix: current Intervention catches filter errors and skips buildings; test accordingly

Minimal LLM Workflow Template
1) Identify scenario objective and constraints.
2) Select existing plugins first (reuse > new code).
3) Install bundles and build scenario template defaults.
4) Create intervention with string refs.
5) Run simulate and inspect metrics/state.
6) If capability gap exists, implement one focused plugin + tests + docs.
